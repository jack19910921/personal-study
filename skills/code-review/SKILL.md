# /code-review

结构化审查代码变更，聚焦安全、性能、正确性、可维护性。根据变更文件自动检测语言，加载语言专属检查清单。

## 用法

```
/code-review                    # 审查当前本地变更（staged + unstaged）
/code-review <PR URL>           # 审查远程 PR
/code-review <PR #123>          # 审查远程 PR（简短写法）
/code-review path/to/file.ts    # 审查指定文件
/code-review "review before I commit"  # 审查本地变更，提交前检查
```

如果未提供任何参数，询问用户要审查什么。

## 工作流

```
┌──────────────────────────────────────────────────────────────┐
│                      CODE REVIEW                              │
├──────────────────────────────────────────────────────────────┤
│  输入层                                                       │
│  ✓ PR URL / PR #     → 拉取 diff，读描述和已有评论            │
│  ✓ 本地变更           → git diff (working + staged)           │
│  ✓ 指定文件           → 读文件内容 + git blame / diff         │
├──────────────────────────────────────────────────────────────┤
│  语言检测                                                     │
│  根据文件扩展名识别语言，加载对应专属检查清单                    │
│  .go → Go  | .py → Python  | .ts/.js → TS/JS                   │
│  .java → Java  | .rs → Rust  | .c/.h/.cpp → C/C++              │
│  .rb → Ruby  | .sh → Shell  | Dockerfile / Makefile → Config   │
├──────────────────────────────────────────────────────────────┤
│  审查维度 = 通用六大支柱 + 语言专属检查                         │
│  ① 安全性       OWASP Top 10 + 语言特有漏洞模式               │
│  ② 性能         通用性能问题 + 语言特有陷阱                    │
│  ③ 正确性       边界条件 + 语言类型系统特性                    │
│  ④ 可维护性     命名、单一职责、重复度、测试覆盖               │
│  ⑤ 可读性       格式一致性、注释充分度                         │
│  ⑥ 测试充分性    框架适配覆盖 + 边界条件建议                   │
├──────────────────────────────────────────────────────────────┤
│  输出层                                                       │
│  ✓ 总览摘要 + 检测到的语言                                     │
│  ✓ Mermaid 变更影响图                                         │
│  ✓ 关键问题表（严重级别 + 文件 + 行号）                         │
│  ✓ 改进建议表（分类 + 文件 + 行号 + 代码示例）                  │
│  ✓ 做得好的地方                                               │
│  ✓ 审查结论（Approve / Request Changes / Needs Discussion）    │
└──────────────────────────────────────────────────────────────┘
```

### 步骤一：确定审查目标

| 输入类型 | 动作 |
|----------|------|
| PR URL / #N | `gh pr checkout <N>` 拉取分支；读 PR 描述理解意图 |
| 本地变更 | `git status` → `git diff` + `git diff --staged` |
| 指定文件 | 读文件 + `git log --oneline -5 -- file` 了解近期变更 |

### 步骤二：上下文准备

1. **理解变更意图**：PR 描述、commit message、关联 issue
2. **检查自动化流水线**：如果项目有 `npm run preflight` / `make test` / `go test` 等，询问是否先跑
3. **语言检测**：根据变更文件的扩展名识别语言，加载对应专属检查清单
4. **定位变更范围**：哪些文件改了、涉及哪些模块、是否热路径

### 步骤三：深度分析（通用 + 语言专属）

#### 通用审查维度（所有语言适用）

##### ① 安全性
- SQL 注入、XSS、CSRF
- 认证和授权缺陷
- 代码中的密钥 / 凭证
- 不安全的反序列化
- 路径遍历、SSRF
- 权限提升可能

##### ② 性能
- N+1 查询
- 不必要的内存分配
- 热路径中的算法复杂度（O(n²) 等）
- 缺失数据库索引
- 未限制的查询或循环
- 资源泄漏（文件句柄、连接池）

##### ③ 正确性
- 边界条件（空输入、null、溢出）
- 竞态条件和并发问题
- 错误处理和传播
- 差一错误
- 类型安全
- 状态机完整性

##### ④ 可维护性
- 命名清晰度
- 单一职责原则
- 代码重复
- 测试覆盖度
- 非显而易见的逻辑是否有注释/文档

##### ⑤ 可读性
- 代码格式是否与项目风格一致
- 函数/方法长度是否合理
- 嵌套深度是否过深
- 变量作用域是否最小化

##### ⑥ 测试充分性
- 新/改代码是否有对应测试
- 现有测试是否覆盖了边界条件
- 建议补充的测试用例（具体到场景）

#### 语言专属检查清单

> 根据检测到的语言，在通用审查基础上叠加以下专属检查项。

##### Go（`.go`）

| 维度 | 检查项 |
|------|--------|
| 安全 | `fmt.Sprintf` 中的用户输入、未验证的 JWT、`exec.Command` 命令注入 |
| 性能 | goroutine 泄漏（未 close channel / 无 select timeout）、未使用 `sync.Pool` 复用对象、大对象传递未用指针 |
| 正确性 | `defer` 在循环中使用、`range` 迭代变量地址捕获、`err` 被忽略未检查、`time` 比较未用 `After/Before` |
| 可读性 | 函数是否遵循 Go 命名习惯（无 Get 前缀等）、错误返回是否用 `fmt.Errorf` 带 `%w` |
| 测试 | 是否使用 table-driven tests、`go test -race` 检查数据竞争 |

##### Python（`.py`）

| 维度 | 检查项 |
|------|--------|
| 安全 | `eval/exec/pickle` 反序列化注入、`subprocess` shell=True、模板渲染未转义 |
| 性能 | N+1 查询（ORM）、全局锁竞争（`threading.Lock`）、未用生成器处理大数据、`+` 拼接大量字符串 |
| 正确性 | 可变默认参数（`def f(x=[])`）、`__init__` 中类变量 vs 实例变量、异常捕获 `except:` 太宽泛 |
| 可读性 | 类型注解是否完整（PEP 484）、docstring 是否符合项目风格 |
| 测试 | 是否使用 `pytest` fixture、mock 外部依赖、参数化测试 |

##### JavaScript / TypeScript（`.js` / `.ts`）

| 维度 | 检查项 |
|------|--------|
| 安全 | `innerHTML` XSS、`localStorage` 存敏感信息、CORS 配置过宽、依赖包已知 CVE |
| 性能 | 未批处理的 DOM 操作、未用 `useMemo/useCallback` 导致重复渲染、未用 `key` 导致 React diff 异常、大对象 deep clone |
| 正确性 | `==` vs `===`、未处理 `undefined/null`、Promise 未 catch、异步函数未 await、闭包捕获过期值 |
| 可读性 | TS 类型是否充分（避免 `any`）、接口命名一致、导出入口是否清晰 |
| 测试 | 是否用 Jest/Vitest mock、覆盖边界条件、snapshot 测试是否合理 |

##### Java（`.java`）

| 维度 | 检查项 |
|------|--------|
| 安全 | 反序列化（`ObjectInputStream`）、LDAP/SQL 注入、不安全的 `javax.crypto` 用法 |
| 性能 | 循环内创建对象、Stream API 在热路径中的开销、连接池未配置上限、N+1 查询（JPA/Hibernate） |
| 正确性 | `equals/hashCode` 一致性、`NullPointerException` 防护、`volatile` 正确使用、事务边界是否合理 |
| 可读性 | 是否符合命名规范、`@Override` 是否标注、异常是 checked 还是 runtime 是否合理 |
| 测试 | JUnit 5 测试、mock 外部服务、集成测试是否启动真实容器 |

##### Rust（`.rs`）

| 维度 | 检查项 |
|------|--------|
| 安全 | `unsafe` 块必要性、`unwrap/expect` 在生产代码中的使用、整数溢出（未用 `wrapping_` / `checked_`） |
| 性能 | `clone()` 是否必要、`Box` vs 栈上分配、`Arc<Mutex<T>>` 锁粒度、迭代器 vs 索引循环 |
| 正确性 | `Send/Sync` 正确性、死锁风险、`Drop` 实现副作用、`async` 中阻塞操作 |
| 可读性 | trait 边界是否清晰、`Result` 错误类型是否具体、文档注释 `///` 覆盖 |
| 测试 | 单元测试 + 集成测试、`cargo test` 覆盖、doc tests 是否通过 |

##### C / C++（`.c` / `.h` / `.cpp` / `.hpp`）

| 维度 | 检查项 |
|------|--------|
| 安全 | 缓冲区溢出（`strcpy/sprintf`）、格式化字符串漏洞、UAF（use-after-free）、双重释放 |
| 性能 | 动态分配在热路径中、缓存未命中（数据局部性）、虚函数调用开销 |
| 正确性 | 未初始化变量、整数溢出、内存泄漏（`malloc` 无对应 `free`）、空指针解引用 |
| 可读性 | RAII 原则（C++）、`const` 正确性、智能指针 vs 裸指针 |
| 测试 | 单元测试覆盖、Valgrind/ASan 检查、边界值测试 |

##### Ruby（`.rb`）

| 维度 | 检查项 |
|------|--------|
| 安全 | SQL 注入（未用参数化查询）、mass assignment、`eval` 动态执行 |
| 性能 | N+1 查询（ActiveRecord）、内存膨胀（大对象未释放）、循环内数据库查询 |
| 正确性 | `nil` 安全访问（`&.`）、`rescue` 范围过宽、线程安全（类变量） |
| 可读性 | Ruby 惯用法（`map` vs `each`）、方法长度、RSpec describe/context 结构 |
| 测试 | RSpec 覆盖、factory_bot 用法、stub/mock 合理性 |

##### Shell（`.sh`）

| 维度 | 检查项 |
|------|--------|
| 安全 | 未引号变量导致注入、`set -e` 缺失导致错误传播中断、硬编码凭证 |
| 正确性 | `$$` vs `$!` 进程 ID 混淆、管道中错误码丢失（需 `set -o pipefail`） |
| 可读性 | 函数封装、参数校验、错误信息是否清晰 |

##### 配置 / IaC（`Dockerfile` / `Makefile` / `*.yml` / `*.yaml` / `*.toml`）

| 维度 | 检查项 |
|------|--------|
| 安全 | Docker 以 root 运行、硬编码密钥、过期基础镜像、权限过宽 |
| 性能 | Docker 层缓存未优化、多阶段构建缺失导致镜像过大 |
| 正确性 | Makefile 中 `.PHONY` 缺失、YAML 缩进错误风险、版本锁缺失 |

### 步骤五：清理（仅远程 PR）

审查完成后，询问用户是否要切回默认分支（main / master）。

## 严重级别定义

| 级别 | 含义 | 示例 |
|------|------|------|
| 🔴 Critical | 必须修复，否则可能导致安全事故 / 数据损坏 / 线上故障 | SQL 注入、凭证硬编码、goroutine 泄漏 |
| 🟡 Warning | 应该修复，可能影响性能或可维护性 | N+1 查询、缺少错误处理、过度嵌套 |
| 🔵 Suggestion | 建议改进，不修复也不会出大问题 | 命名不够清晰、可以提取函数、缺少注释 |
| 💡 Nitpick | 格式 / 风格小问题 | 多余空行、命名风格不统一 |

## 语气指南

- **建设性**：解释「为什么」要这样改，而非只说「不好」
- **专业友好**：对做得好的地方也要肯定
- **代码证据**：每个问题都要引用实际文件和行号，证明你真的读了代码
- **优先级排序**：先 🔴 再 🟡，避免 nitpick 淹没关键问题
- **避免假设**：不要说「这里应该有 bug」而不看实际代码

## 提示

1. **提供上下文** — 「这是热路径」或「这里处理 PII 数据」有助于聚焦审查
2. **指定关注点** — 「重点关注安全性」或「检查并发安全」可以缩小范围
3. **附带测试** — 我会同时检查测试覆盖和质量
4. **提交前审查** — 在 commit 之前运行，避免把问题带入历史
